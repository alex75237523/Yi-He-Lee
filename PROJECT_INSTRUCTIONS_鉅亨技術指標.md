# 專案固定指令：鉅亨網台股技術指標每日擷取

本文件為此專案所有 AI、Codex、Claude Code 與人工開發的強制規範。開始任何修改前，必須先閱讀目前程式碼、資料庫結構、既有排程與文件，再依本文件執行；不得未確認現況就直接重寫或更換技術棧。

## 一、每日資料擷取需求

### 1. 固定執行時間

- 時區一律使用 `Asia/Taipei`。
- 每日台北時間 `13:35`，台股收盤後執行。
- 排程時間、查詢日期、資料日期與紀錄時間不得使用伺服器未轉換的 UTC 日期。
- 必須由程式明確取得台北當地日期，例如 `targetDate`，不可硬編日期。

### 2. 固定資料來源

每日必須抓取以下兩個網址的完整 List 清單：

1. `https://www.cnyes.com/twstock/a_technical4.aspx`
   - 資料類型：股價多頭排列。
2. `https://www.cnyes.com/twstock/a_technical5.aspx`
   - 資料類型：股價空頭排列。

兩個頁面都必須抓取：

- 集中市場。
- 店頭市場。

不得只抓預設市場、畫面第一批資料、前 50 筆或部分清單。必須確認該日期、該市場的表格所有資料列都已完成擷取。

### 3. 當日日期強制規則

- 「設定查詢」日期一定要使用排程執行當日的台北日期。
- 頁面顯示的資料日期必須與 `targetDate` 完全一致。
- 請求日期、頁面日期、資料寫入的交易日期必須一致。
- 絕對禁止因網站尚未更新、休市、例假日或錯誤而自動改抓前一交易日。
- 絕對禁止把前一天資料寫成今天。
- 在頁面日期尚未等於當日以前，不得寫入正式資料表。
- 若 13:35 時網站仍顯示前一交易日，必須記錄「網站尚未更新當日資料」並依既有排程重試機制重試。
- 若為休市日或當日確實沒有資料，應記錄明確狀態，不得用舊資料補齊。

### 4. 擷取欄位

每一筆至少保存以下資料：

- 查詢／交易日期。
- 資料類型：`BullishAlignment`（多頭排列）或 `BearishAlignment`（空頭排列）。
- 市場別：集中市場或店頭市場。
- 股票代碼。
- 股票名稱。
- 收盤價。
- 5 日均價。
- 20 日均價。
- 60 日均價。
- 120 日均價。
- 來源網址。
- 擷取開始時間。
- 擷取完成時間。
- 建立時間與最後更新時間。

數值欄位應使用適合的 Decimal 型別，不得以顯示文字永久保存為唯一資料來源。

### 5. 完整性與日期驗證

寫入前必須完成以下檢查：

- HTTP 回應成功。
- 頁面不是錯誤頁、驗證頁或不完整 HTML。
- 頁面資料類型與來源網址相符。
- 市場別與本次查詢條件相符。
- 頁面顯示日期等於 `targetDate`。
- 表頭結構存在，欄位可正確對應。
- 每筆股票代碼、名稱與必要數值格式有效。
- 完整讀取所有資料列後，才可進入資料庫交易。
- 空清單必須能區分「合法零筆」與「解析失敗／網頁尚未更新」，不可直接視為成功。

任何一項驗證失敗，都不得把該批次標記為成功。

## 二、資料庫與重跑規則

### 1. 關聯式與去重設計

- 資料庫必須採關聯式設計，避免重複保存股票基本資料。
- 股票基本資料應以股票代碼或既有股票主檔關聯，不得因每日擷取重複新增相同股票名稱。
- 建議資料唯一鍵至少包含：
  - `TradeDate`
  - `IndicatorType`
  - `MarketType`
  - `StockCode`
- 排程必須具備冪等性，重跑同一天、同資料類型、同市場時不得產生重複資料。
- 寫入採用安全的 Upsert、先比對後更新，或符合現有架構的等效做法。
- 同一批次應使用資料庫交易，避免只寫入部分市場或部分資料列。

### 2. 資料表與欄位

- 新增的每個資料表與欄位都必須有繁體中文註解。
- 不得為方便而將整份頁面資料塞入單一 JSON 欄位取代正常關聯式設計。
- 大量資料處理可使用暫存表、中繼表或快取表，但必須有清除與失敗復原機制。
- SQL 必須參數化，不得拼接外部網頁內容或使用者輸入。
- `ALTER`、`UPDATE`、`DELETE` 必須說明條件與影響範圍。

## 三、程式架構規範

### 1. 分層責任

依目前專案技術棧與既有架構實作，不得為此功能任意重寫整個專案。C# 專案須遵守 MVC／三層架構精神：

- 排程層：只負責在 13:35 觸發工作，不放網頁解析與資料庫商業邏輯。
- Service／Business Layer：負責日期規則、流程控制、重試判斷、批次驗證、交易一致性與 Upsert 規則。
- Crawler／Provider／Adapter：只負責請求指定來源、切換日期與市場條件、取得原始內容。
- Parser：只負責將頁面表格轉成結構化資料，並回報解析錯誤。
- Repository／DAO：只負責 CRUD、Query、參數化 SQL 與資料轉換。
- Controller：若有管理介面，只處理 HTTP request、Claims、呼叫 Service 與回傳 View／JSON。
- ViewModel／ApiModel／Entity 不得混用職責。
- 前端只負責查詢畫面、狀態與操作回饋，不得成為唯一的日期或資料驗證來源。
- 後端 Service 必須重新驗證所有重要規則。

### 2. 共通程式規則

- 使用物件導向、低耦合、高內聚。
- 可重複使用規則放在 Service 或 Helper，不得複製到多個排程、Controller、View 或 JavaScript。
- 使用既有 HttpClient、排程、Logging、DI 與設定機制；沒有充分理由不得另外導入重型框架。
- 網址、逾時、重試條件與排程時間應設定化，但上述兩個固定來源與「當日日期」規則不得被任意關閉。
- HTTP 呼叫必須設定合理 Timeout、CancellationToken、錯誤處理與識別用 User-Agent。
- 不得繞過網站安全機制、驗證或存取限制。
- 所有重要程式碼必須有繁體中文註解。

## 四、排程執行紀錄

每次排程至少記錄：

- Job／Batch ID。
- 台北執行日期與時間。
- 目標日期。
- 來源網址。
- 資料類型。
- 市場別。
- 頁面實際顯示日期。
- 抓取筆數。
- 新增筆數。
- 更新筆數。
- 略過筆數。
- 失敗筆數。
- 開始與完成時間。
- 執行狀態。
- 錯誤訊息與重試次數。

成功條件必須是兩個網址、兩種市場均完成，且所有頁面日期皆等於當日。不得因其中一頁成功就把整批標記為成功。

## 五、測試與驗收

至少涵蓋以下測試：

- 多頭排列頁面解析。
- 空頭排列頁面解析。
- 集中市場與店頭市場解析。
- 當日日期正確時可寫入。
- 頁面仍為昨日日期時拒絕寫入。
- 休市或合法零筆資料的判斷。
- HTML 欄位順序改變或缺欄時能失敗並留下紀錄。
- 同一天排程重跑不會新增重複資料。
- 某一市場失敗時整批交易與狀態正確。
- 時區跨日、UTC 與台北日期不同時仍使用台北日期。

修改後應執行現有 Solution 的 Build 與相關測試。無法執行時，必須說明原因並提供人工驗證步驟。

若功能流程、狀態碼、資料表欄位、API request／response、前端操作流程、權限規則或排程規則異動，必須更新對應 Markdown 文件與異動紀錄。不得刪除歷史需求；過期內容應標記為歷史版本。

文件、重要註解與完成回報一律使用繁體中文。
