# SQLite 資料庫結構

實際 Schema：`database/schema.sql`，程式內建相同建表語句。

- `Stocks`：股票主檔，避免重複名稱。
- `TechnicalIndicatorDaily`：每日技術指標；唯一鍵為交易日期、排列類型、市場、股票代碼。
- `JobRuns`：每次排程主紀錄。
- `JobRunDetails`：來源與市場的抓取明細、頁面日期、筆數及錯誤。
- `CustomerHoldingSnapshots`：每日有效持股快照。2026-07-09 起 `EntryAveragePrice`（舊版單一欄位）曾改名為 `CurrentPrice`（Excel「現價」欄位，外部 DDE）並允許 `NULL`（現價無法判讀時）；程式啟動時自動重建遷移舊資料庫，舊列數值原樣保留（當時的比較基準為進場價，僅屬歷史紀錄）。同日另新增 `CurrentPriceIssue`（TEXT NULL）欄位，保存 DDE 現價無效原因（例如「#N/A（DDE 尚未取得資料，看盤軟體可能未開啟或未連線）」），以 `ALTER TABLE ADD COLUMN` 安全新增，不刪除或覆蓋既有資料列；`CurrentPrice` 為 `NULL` 不得因此刪除或略過該筆持股快照。
  **2026-07-11 恢復雙價格判斷**：新增真正獨立的 `EntryAveragePrice`（NUMERIC NULL，Excel「進場價/平均價」欄位，非 DDE）與 `EntryAveragePriceIssue`（TEXT NULL，無效原因）兩個欄位，以 `ALTER TABLE ADD COLUMN` 安全新增。**注意**：欄位名稱 `EntryAveragePrice` 與 2026-07-09 之前的舊版歷史欄位同名但意義完全不同；`SqliteYiHeLeeRepository.MigrateEntryAveragePriceToCurrentPriceAsync` 已加上「`CurrentPrice` 是否已存在」的守門檢查，確保舊版一次性搬移邏輯不會被新版同名欄位誤觸發而覆蓋 `CurrentPrice`。`CurrentPrice` 與 `EntryAveragePrice` 是兩個完全獨立的價格來源，禁止共用同一個 Issue 或同一個數值。
- `StrategyAlerts`：均線觸發、進場價/平均價無效、現價無效或缺少技術資料清單。`AlertKind`：1 均線觸發、2 缺少技術資料、3 現價無效（DDE）、4 進場價/平均價無效（2026-07-11 新增，非 DDE 欄位）。`CurrentPrice`／`CurrentPriceIssue`／`EntryAveragePrice`／`EntryAveragePriceIssue` 四欄分別保存兩個價格與各自的無效原因，禁止互相覆蓋或代替。**唯一鍵新增 `AlertKind`**（`UNIQUE (TradeDate, WorkbookPath, SheetName, ExcelRow, StockCode, AlertKind)`）：因同一持股現在可能同時產生「進場價/平均價異常」與「現價異常」兩筆通知，舊唯一鍵（不含 `AlertKind`）會讓後寫入的通知覆蓋先寫入的通知；SQLite 無法直接修改既有 `UNIQUE` 約束，`MigrateStrategyAlertsAddEntryAveragePriceAndUniqueKeyAsync` 採「改名舊表→由 SchemaSql 建新表→搬移→刪舊表」方式安全重建，舊資料列原樣搬移，新欄位為 `NULL`。

同日重跑時，正式技術批次會先刪除同日期／類型／市場的舊快照，再在同一交易插入完整新清單；持股與通知也會依同日期及活頁簿重建，避免舊資料殘留。

## 官方每日收盤價與均線（TWSE／TPEx）

2026-07-09 新增。均線策略正式數值改由官方收盤價計算，鉅亨網 `Stocks`／`TechnicalIndicatorDaily` 僅作多頭／空頭排列清單保存與交叉驗證。

- `StockMaster`：官方股票主檔（TWSE=上市／TPEx=上櫃），依 `StockCode` 去重；`MarketType` 由當日實際出現在哪個官方來源決定並自動更新。
- `StockDailyPrice`：官方每日收盤價，唯一鍵 `StockId + TradeDate`；`SourceDataDate` 為來源實際回報的資料日期，寫入前必須等於 `TradeDate`。
- `StockMovingAverage`：均線計算結果快取，唯一鍵 `StockId + TradeDate`；任一均線因有效交易日數不足而為 `NULL` 時，`CalculationStatus` 記為 `InsufficientHistory`。
  - 2026-07-11 修正：每日五日均價策略前置作業以當日 `StockDailyPrice` 已保存的官方收盤價股票全集計算並保存 `StockMovingAverage`，不以客戶持股、Excel「現價」或 DDE 狀態決定計算範圍。WinForms「均價資料異常」頁籤只查詢 `CalculationStatus != 1` 的列，顯示代碼、名稱、收盤價、MA5／20／60／120 與缺少原因；客戶比對只能作為下游讀取這些已保存資料。
  - 2026-07-11 新增：`CalculationStatus` 擴充為 `1正常／2交易日數不足／3當日收盤價缺失／4歷史回補失敗`（既有數值 1／2 意義不變，屬向下相容擴充，無需 CHECK 限制異動）；新增 `LatestAvailableTradeDate`（TEXT NULL，最新一筆已知有效收盤價日期）與 `MissingReason`（TEXT NULL，缺少原因說明文字），供 WinForms「均價資料異常」頁籤說明異常。Excel「每日五日均價策略」只輸出代碼、名稱、收盤價與 MA5／MA20／MA60／MA120，不輸出診斷欄位。既有資料庫以 `PRAGMA table_info` 檢查後 `ALTER TABLE ADD COLUMN` 安全新增，不刪除或覆蓋既有資料列。
  - 2026-07-11 起，`AvailableTradingDayCount`／`CalculationStatus` 之判斷改為**逐檔股票**計算：歷史回補完成判斷不再只看市場整體交易日數，個別股票即使市場整體已足夠，仍會依自身實際筆數判斷是否需要繼續回補。詳見 `docs/02_架構與資料流程.md`「逐檔歷史完整性驗證」與 `PROJECT_INSTRUCTIONS_鉅亨技術指標.md`。
- `OfficialPriceBatch`：官方價格批次紀錄，`JobType` 區分「每日排程(DailyMarketData)」與「歷史回補(HistoricalBackfill)」，兩者分開記錄、互不混用不同日期資料；`Status` 至少涵蓋 `Pending／Running／NotPublished／Holiday／Succeeded／PartialFailed／Failed／InsufficientHistory`。
- `EmergingHistoricalNoDataProbe`（2026-07-11 新增）：`StockCode` + `TradeDate` 唯一鍵，記錄興櫃歷史回補「已查詢但確認查無資料」的組合。興櫃很多持股是最近才開始交易，回補迴圈往回走到更早的日期時官方端點永遠查不到資料；沒有這張表時，這個「查不到」的結果不會被記住，導致每次執行都要對同一批股票、同一批過去日期重新發送請求。歷史資料一經確認即不會改變，記錄後未來執行可直接略過，對應 `IMarketDataRepository.GetConfirmedNoEmergingDataCodesAsync`／`RecordConfirmedNoEmergingDataAsync`。詳見 `docs/05_異動紀錄.md` 對應項目。
- `HistoricalBackfillExhaustionProbe`（2026-07-11 新增）：記錄「同一策略日期、同一 MA 所需交易日數、同一最大回看日曆日數、同一持股組合」已走完整個歷史回補範圍但仍不足的結果。用途是避免同一指定日期重跑時，對近期掛牌、停牌或歷史來源確認查無資料的持股反覆從 `targetDate - 1` 往前掃描。唯一鍵為 `TargetDate + RequiredTradingDays + MaxLookbackCalendarDays + StockSetKey`；僅在官方來源沒有 `Failed`／`NotPublished` 等暫時性狀態時寫入。Service 仍會先即時查 DB 逐檔筆數，已補足時不受本表阻擋。

同日重跑時，`StockDailyPrice` 依 `StockId + TradeDate` Upsert（更新既有列，不新增重複列）；`OfficialPriceBatch` 已有成功批次時，`MarketPriceService` 會直接略過重複抓取（冪等）。歷史回補一律保存原始 `TradeDate`，不得改寫為執行當日；`targetDate` 當日資料一律由每日正式排程負責，兩者職責分離。

## 歷史收盤價查詢畫面／使用者手動回補（2026-07-09 新增）

新增功能：使用者可在「歷史收盤價」畫面手動觸發回補（預設最近 5 個有效交易日、可調整），並即時查看整體與工作明細進度。設計原則是**重用**既有官方收盤價抓取／驗證／Upsert 機制，只新增「使用者觸發批次的進度追蹤」與「查詢分頁」兩塊：

- `StockDailyPrice` 額外新增（`ALTER TABLE ADD COLUMN`，相容既有資料庫）：`OpenPrice`／`HighPrice`／`LowPrice`／`TradeVolume`／`TradeValue`／`TransactionCount`／`PriceChange`（皆為選填，目前 TWSE／TPEx Provider 僅穩定解析收盤價，其餘欄位保持 `NULL`，不得偽造）、`IsOfficial`（固定為真，本表僅保存官方來源）。
- `StockPriceImportJob`：使用者手動觸發的回補批次主紀錄（一次「立即回補」一筆），欄位對應畫面「整體進度」需求（`TotalTasks`／`CompletedTasks`／`SuccessTasks`／`FailedTasks`／`SkippedTasks`／各筆數統計／`ProgressPercent`／`Status`）。
- `StockPriceImportTask`：一個工作＝一個市場＋一個交易日期，唯一鍵 `JobId + MarketType + RequestedDate`；`ActualTradeDate` 為官方回報的實際交易日期，可能與 `RequestedDate`（請求日期）不同。實際抓取／解析／驗證／寫入 `StockDailyPrice` 一律呼叫既有 `IMarketPriceService.FetchAndSaveSingleAsync`（由既有 `FetchAndSaveOneAsync` 重構為公開方法），本表只記錄進度，不重複實作抓取邏輯。
- `CnyesCrossValidation`：鉅亨網多頭／空頭排列與官方自算均線的交叉驗證紀錄，唯一鍵 `TradeDate + MarketType + StockCode + WindowDays`；僅作驗證追查，不得覆蓋或取代官方資料。

批次與工作進度皆彙總自 `StockPriceImportTask` 明細列（不使用記憶體計數器），因此使用者重新整理／重新開啟畫面後，仍可由資料庫回復目前進度。索引：`StockPriceImportJob(Status, CreatedAt)`、`StockPriceImportTask(JobId, Status)`、`StockMaster(MarketType, StockCode)`、`CnyesCrossValidation(TradeDate)`。

歷史收盤價查詢分頁（`IMarketDataRepository.QueryDailyPricesAsync`）以單一 SQL（Window Function：`AVG(...) OVER (PARTITION BY StockId ORDER BY TradeDate ROWS BETWEEN N-1 PRECEDING AND CURRENT ROW)`）計算 MA5／MA20／MA60／MA120 並支援市場別／代碼或名稱關鍵字／日期區間篩選與分頁，避免對每一列個別查詢前 N 筆造成 N+1。
