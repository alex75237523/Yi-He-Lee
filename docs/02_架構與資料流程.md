# 架構與資料流程

```text
DailyScheduleCoordinator
  -> DailyJobService
     -> DailyMarketDataJob -> MarketPriceService -> ITwseMarketDataProvider／ITpexMarketDataProvider
        -> SqliteMarketDataRepository（官方每日收盤價，來源日期驗證後單一交易 Upsert）
     -> ExcelWorkbookService（讀取持股）
     -> StockIdentityResolutionService（逐檔正規化、前導零回復與官方主檔身分驗證，決定最終使用代碼與市場別）
     -> HistoricalBackfillJob -> MarketPriceService（best-effort 回補 MA120 所需歷史資料，保留原始 TradeDate；
        上市／上櫃／興櫃皆依持股代碼逐檔確認是否足夠，不再只看市場整體交易日數）
     -> MovingAverageService -> SqliteMarketDataRepository（依官方收盤價計算並快取 MA5／20／60／120）
     -> CrawlerRegistry -> CnyesTechnicalAlignmentCrawler -> Parser
     -> SqliteYiHeLeeRepository（鉅亨多頭／空頭清單，四個必要批次驗證後單一交易；僅作清單保存與交叉驗證；
        本步驟為 best-effort：失敗或網站尚未更新時只記錄提醒訊息，不阻擋已由官方價格算出的正式均線與策略流程）
     -> StrategyEvaluationService.Evaluate（中央通知子集合：觸發／現價無效／無法判斷，寫入 StrategyAlerts 資料表）
     -> StrategyEvaluationService.EvaluateAll（每一筆有效持股皆產生一筆 HoldingStrategyResult，不因未觸發而略過；
        DDE 無效只影響 OverallResult，官方收盤價與均線欄位照常帶入）
     -> SqliteYiHeLeeRepository（持股快照及通知，CustomerHoldingSnapshots.CurrentPriceIssue 保存 DDE 錯誤原因）
     -> ExcelWorkbookService（覆寫輸出頁籤並儲存，25 欄含客戶／DDE狀態／DDE錯誤原因／MA5～MA120比較／整體結果／
        觸發條件／鉅亨驗證狀態／計算時間；列數＝全部有效持股數＋診斷列數，不等於 alerts 數量）
     -> WinFormsUserInteraction -> 中央通知視窗（仍依 StrategyAlert 子集合顯示，行為不變）
```

## DDE 現價異常隔離（2026-07-11 新增）

`CustomerHolding.CurrentPrice` 為 `null`（DDE 無效）只影響 `StrategyEvaluationService` 最後一步「現價 vs 均線」比較的結果文字（`OverallResult` 顯示為「現價無效，暫時無法判斷」），不得影響：持股辨識與股票代碼正規化、官方收盤價擷取與 SQLite 寫入、逐檔歷史回補、MA5／MA20／MA60／MA120 計算、鉅亨網交叉驗證、Excel 輸出（該持股仍完整列出一列，均線欄位照常顯示）。`DailyJobService.RunAsync` 讀取 Excel 持股後才知道各筆 DDE 是否有效，而官方收盤價擷取、歷史回補、均線計算等步驟本身皆不以 `CurrentPrice` 作為輸入或先決條件，因此即使某筆持股 DDE 無效，也不會影響同一批次其他步驟或其他持股。`tests/YiHeLee.Tests/DailyJobServiceTests.cs` 以 Fake／Spy 建構完整依賴鏈驗證此行為並防止回歸。

## 逐檔歷史完整性驗證（2026-07-11 新增）

修正「市場整體交易日數已足夠、但個別股票仍不足」（例如上櫃市場整體已有 120 日，但 5351 鈺創、8299 群聯、3691 碩禾等個股只有 83／76／65 筆）導致只有 MA5、缺少 MA20／60／120 的問題：

- `StockIdentityResolutionService`（`YiHeLee.Application/Services/`）：Excel 讀出的原始股票代碼先經
  `StockCodeNormalizer` 正規化，再以官方股票主檔（`IMarketDataRepository.GetStockMarketTypesAsync`）
  驗證身分；查無資料且為 1～3 碼純數字時，嘗試補零為 4／5 碼後再查一次官方主檔，**只有補零後代碼確實
  存在於官方主檔才採用**，不盲目補零。回傳結果同時標示是否納入均線策略（`StockIdentityResolver` 依
  台股實際商品格式判斷，權證明確排除）。
- `MarketPriceService.BackfillHistoryAsync` 完成判斷改為「市場整體交易日數足夠」**且**「傳入的持股
  逐檔皆已達 `RequiredTradingDaysForMa120`」兩者皆成立才停止；回補本身仍是整批市場下載（效率不變），
  只有「是否已完成」的判斷改為逐檔確認。
- `IMarketDataRepository.GetDistinctTradeDateCountAsync(stockCode)`（既有方法，先前僅用於興櫃逐檔
  回補）擴大套用到上市／上櫃逐檔檢查，不需新增 Repository 方法。
- 回補批次若出現 `Failed` 狀態（官方來源逾時或格式異常），`DailyJobService` 會把該市場仍
  `InsufficientHistory` 的股票均線結果升級為 `CalculationStatus.BackfillFailed` 並附上原因，與「單純
  歷史資料不足（可能為近期上市）」明確區分，皆反映在 Excel 輸出「計算狀態」「缺少原因」欄位。

## 官方每日收盤價來源分層

- `ITwseMarketDataProvider`／`TwseMarketDataProvider`、`ITpexMarketDataProvider`／`TpexMarketDataProvider`：只呼叫官方端點、轉成 DTO、回報來源資料日期與解析錯誤，不含日期驗證或 SQL。
- `IMarketPriceService`／`MarketPriceService`：台北日期與來源日期驗證、休市／尚未公布判斷、冪等 Upsert、批次狀態記錄、歷史回補協調。
- `IMovingAverageService`／`MovingAverageService`：依有效交易日計算 MA5／MA20／MA60／MA120，資料不足時對應欄位為 `null`。
- `DailyMarketDataJob`／`HistoricalBackfillJob`：排程觸發用的薄封裝，只呼叫 `IMarketPriceService`，不含 HTTP、HTML／JSON 解析或 SQL。
- `IMarketDataRepository`／`SqliteMarketDataRepository`：`StockMaster`／`StockDailyPrice`／`StockMovingAverage`／`OfficialPriceBatch` 的參數化 SQL、交易與 Upsert。

## 重試層級

- 爬蟲短重試：網路、瀏覽器、DOM 暫時錯誤。
- Excel 短重試：RPC 呼叫被拒、Excel 忙碌、正在編輯儲存格。
- 排程長重試：網站尚未更新當日資料、短重試耗盡等可重試狀態。
- 設定無效、必要 Provider 不存在、Excel 唯讀／受保護檢視、輸出頁籤受保護等不可重試錯誤，需要人工修正。
- 非必要來源必須集中／店頭兩個市場都成功才會納入正式批次；其中一個市場失敗時整個非必要來源略過。
- 2026-07-11 更新：每日流程改為先處理官方收盤價，再讀取 Excel 持股代碼（同時顯示正在讀取哪個客戶頁籤），接著依 DB 缺口回補 MA120 所需歷史收盤價、計算 MA5／MA20／MA60／MA120，最後才擷取鉅亨網多頭／空頭排列清單作保存與交叉驗證。鉅亨網不再位於流程最前面，也不作正式均線來源。
- 2026-07-11 更新：TPEx 上櫃歷史回補改用新版 `afterTrading/otc` 端點與斜線日期格式，避免舊端點或壓縮日期格式靜默回傳最新交易日。興櫃歷史回補改用 TPEx 官方 `emerging/historical` 個股月份端點；因官方只提供「個股＋月份」查詢，程式只對 Excel 持股中已辨識為興櫃且 DB 缺資料的股票逐檔補。
- 2026-07-09 更新：鉅亨網多頭／空頭排列清單擷取改為 best-effort，不再是整批工作成功與否的門檻。原因是均線正式判斷已改依 TWSE／TPEx 官方資料計算，不再依賴鉅亨網數值；鉅亨網尚未更新或擷取失敗時，`DailyJobService` 僅記錄提醒訊息（`JobRunSummary.Message` 附加「提醒：…」字樣、`JobRunDetails` 仍完整記錄失敗細節），不影響已由官方價格算出的正式均線、策略判斷與 Excel 寫入。
- 官方每日收盤價（TWSE／TPEx）：來源資料日期不等於 `targetDate` 時記為 `NotPublished`（可重試，等同「網站尚未更新」）；TWSE 明確回報無資料時記為 `Holiday`。TPEx 端點對非交易日會靜默回傳最近一個交易日資料而非報錯，因此僅在同日 TWSE 已確認休市時，才會把 TPEx 的日期不符情形也記為 `Holiday`，否則一律視為 `NotPublished`，絕不接受用前一交易日資料頂替當日。
- 歷史回補以 DB 為準：若 `StockDailyPrice` 已保存指定市場、指定交易日的正式收盤價，即使沒有對應的歷史回補批次紀錄，也略過該日該市場的重複抓取。

## 歷史收盤價查詢畫面／使用者手動回補（2026-07-09 新增）

```text
TrayApplicationContext（選單「歷史收盤價」）
  -> HistoricalPriceForm（查詢畫面＋抓取進度面板）
     -> 查詢：IMarketDataRepository.QueryDailyPricesAsync（分頁＋SQL Window Function 計算 MA5/20/60/120）
     -> 立即回補：IStockHistoryImportService.CreateJobAsync（快速建立 StockPriceImportJob／Task，皆為 Queued）
        -> IStockHistoryImportService.RunJobAsync（背景執行，不阻塞 UI 執行緒）
           -> Parallel.ForEachAsync（MaxConcurrency 上限，預設4，不超過8）
              -> IMarketPriceService.FetchAndSaveSingleAsync（單一市場＋單一交易日，重用既有驗證與冪等 Upsert）
              -> 暫時性錯誤指數退避重試（2/4/8秒，經 TransientFailureClassifier 區分是否可重試）
           -> IStockPriceImportRepository（每個工作完成後即時彙總進度，供輪詢與重新整理回復）
     -> 取消抓取：CancellationTokenSource.Cancel -> IStockPriceImportRepository.CancelJobAsync
     -> 與鉅亨網交叉驗證：CrawlerRegistry（既有鉅亨爬蟲）-> IStockPriceValidationService.ValidateAsync
        -> IStockPriceValidationRepository（CnyesCrossValidation，僅供驗證追查，不覆蓋官方資料）
```

- 工作單位固定為「市場＋交易日期」，例如回補 5 個有效交易日、市場範圍全部時，約產生 10 個工作，而非每檔股票各發送一次請求。
- 候選交易日由觸發當下台北日期往前一日開始，排除週六、週日；國定假日等無法事先得知的休市日仍會實際送出請求，由既有官方來源誠實回報並記為 `Holiday`，不計入有效交易日、也不算失敗。
- 進度（`StockPriceImportJob`／`StockPriceImportTask`）皆彙總自資料庫明細列，不依賴記憶體計數器，重新整理畫面或重啟程式後仍可回復目前進度。
- SQLite 並行安全：沿用既有 `SqliteMarketDataRepository` 慣例（每次操作各自開啟連線＋`busy_timeout=10000`＋單一交易），並行只發生在 HTTP 抓取階段，資料庫寫入仍是「每個市場＋交易日完整下載、解析、驗證後才進入單一交易 Upsert」，不會有多個 Task 共用同一個 `DbConnection`。

## 權限

預設與使用者同權限執行，以連接一般權限的 Excel。可從系統匣選單要求 UAC 提升，但提升後 Excel 也必須以相同權限啟動。
