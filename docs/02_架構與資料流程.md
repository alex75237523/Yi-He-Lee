# 架構與資料流程

## 盤中／收盤流程拆分（2026-07-13 現行版本）

系統拆成兩條完全不同的流程，不再放在同一個服務中依時間判斷：

```text
盤中：優先使用上一交易日已保存均價；若基準缺漏，先在同一次呼叫內補齊官方收盤價並重算均價，再讀取客戶價格並判斷。
收盤：抓取今日正式收盤價、換算今日均價、保存 SQLite 與 Excel。
今日收盤產生的均價：下一個交易日盤中才開始使用。
```

```text
MarketWorkflowScheduleCoordinator（時區 Asia/Taipei；決策函式 MarketWorkflowPlanner 可獨立測試）
├─ IntradayMonitoringService（交易日 09:00 ≦ 台北時間 < 13:30，依 IntradayCheckIntervalSeconds 預設每 30 秒一次；
│     程式於盤中啟動時立即判斷一次；上一 Tick 未完成時下一 Tick 直接略過，不排隊）
│   -> TradingDateResolver（解析真正上一交易日：StockDailyPrice／StockMovingAverage／OfficialPriceBatch；
│        禁止 today-1；快照不完整或收盤更新失敗時先交由 BaselinePreparationService 準備，禁止退回更舊均價）
│   -> BaselinePreparationService.EnsureBaselineDataAsync（只在基準缺漏時執行；
│        由既有 SQLite 資料推導 BaselinePreparationState，不新增資料表、不用記憶體 bool 當 Ready 狀態；
│        缺官方收盤價才呼叫 MarketPriceService.BackfillHistoryAsync，已有資料則走冪等快速略過；
│        缺 StockMovingAverage 才呼叫 MovingAverageService.CalculateManyAsync 並保存 MA5／20／60／120；
│        同一 EvaluationDate 同時間只允許一個準備流程，其他 Tick 顯示「正在準備上一交易日資料」並略過）
│   -> TradingDateResolver（準備後重新驗證 BaselineTradeDate；驗證成功後不得直接結束）
│   -> SqliteMarketDataRepository.GetMovingAverageResultsAsync(BaselineTradeDate)（只讀已保存快照，不計算均線）
│   -> ExcelWorkbookService.ReadHoldingsAsync（唯讀客戶頁籤：進場價/平均價＋現價（DDE）；不備份、不儲存、不覆寫）
│   -> StockIdentityResolutionService（股票代碼正規化，只查 DB 主檔）
│   -> StrategyEvaluationService.Evaluate（既有雙價格判斷公式，本次拆分不修改）
│   -> SqliteIntradayStateRepository（IntradayAlertState 通知去重＋IntradayEvaluationRun 每次 Tick 摘要）
│   -> 中央結果畫面（持續更新目前成立清單；只有新觸發或重要錯誤才主動彈出）
│   ※ 快速路徑禁止：TWSE／TPEx／興櫃 Provider、鉅亨擷取、歷史回補、均線計算、StockDailyPrice／StockMovingAverage 寫入。
│      只有基準缺漏時才可執行 BaselinePreparationService；仍禁止 Excel「每日五日均價策略」寫入、
│      以盤中現價推算或取代正式收盤價。
└─ ClosePrecomputeJobService（每日台北時間 13:35；13:35 後啟動且今日未成功時補跑）
    -> DailyMarketDataJob -> MarketPriceService -> ITwseMarketDataProvider／ITpexMarketDataProvider（來源日期嚴格驗證）
       -> SqliteMarketDataRepository（StockDailyPrice 冪等 Upsert）
    -> MarketPriceService.FetchAndSaveSingleAsync（TPEx 興櫃當日行情，best-effort）
    -> MovingAverageService -> SqliteMarketDataRepository（依 DB 既有官方收盤價計算並保存 MA5／20／60／120；
       歷史不足時保存 InsufficientHistory，不自動回補）
    -> CrawlerRegistry -> CnyesTechnicalAlignmentCrawler（鉅亨清單保存與交叉驗證，best-effort）
    -> ExcelWorkbookService.WriteStrategyResultsAsync（覆寫「每日五日均價策略」七欄；寫入前備份）
    -> SqliteYiHeLeeRepository（JobRuns 收盤 Job／錯誤／重試紀錄）
    -> 完成後該交易日標記為「下一交易日盤中判斷基準已準備完成」（BaselineTradeDate）
    ※ 禁止：讀取客戶頁籤現價、盤中監控、呼叫 StrategyEvaluationService；
      不得因 DDE 或進場價異常導致收盤工作失敗

IWorkflowExecutionGate／WorkflowExecutionGate（共用流程協調鎖）
  - 同一時間只能有一個盤中判斷／基準準備；收盤更新執行時不得啟動盤中判斷；
    Excel COM 操作不會同時由兩個流程執行；盤中查詢只讀 SQLite 已完整提交的均價快照。
```

盤中判斷的日期欄位必須明確分開：`EvaluationDate`（今天盤中判斷日期）、`BaselineTradeDate`（使用的上一交易日均價日期）、`EvaluatedAt`（實際判斷時間）；通知、Log、資料庫與畫面都必須能看出「現在判斷的是哪一天、使用的是哪一個交易日的均價」。

同一個 `EvaluationDate`／`BaselineTradeDate` 一旦由 SQLite 推導為 Ready（該基準日已有官方收盤價，且同一批股票皆已有完整 MA5／MA20／MA60／MA120），後續手動第二次盤中判斷、下一個 30 秒 Tick，以及程式重啟後的盤中判斷，都必須直接沿用已保存基準，只重新讀取 Excel 最新進場價／平均價與 DDE 現價並執行策略判斷，不得再次呼叫官方行情 Provider、查詢相同年度官方日曆、歷史回補、重算同日均價或重寫同日收盤價。

`ClosePriceRetryIntervalSeconds`（預設 60 秒，範圍 10～600 秒）只用於 13:35 後官方收盤價尚未公布、HTTP 暫時失敗、來源日期不符等可重試收盤更新情境。不可用它重試 Excel 設定錯誤、工作簿唯讀、受保護檢視、Schema 錯誤或其他需要人工處理的不可重試問題。

> **歷史版本（2026-07-13 前，已由本次拆分取代）**：原本由 `DailyScheduleCoordinator -> DailyJobService.RunAsync`
> 在每日 13:35 的單一流程中依序完成官方收盤價擷取、均線計算、鉅亨清單、Excel 持股讀取（含 DDE 現價）、
> 策略判斷與 Excel 輸出，「每日 13:35 抓資料後立即讀取客戶現價並判斷」「收盤日均價直接用於同一天策略」。
> 此設計已拆分為上述兩條流程；策略比較公式本身（雙價格判斷）不變。

## 雙價格異常隔離（2026-07-11 恢復並強化）

客戶 Excel 持股表同時有「進場價/平均價」（`CustomerHolding.EntryAveragePrice`）與「現價」（`CustomerHolding.CurrentPrice`，DDE）兩個完全獨立的欄位。任一欄位為 `null`（無效）只影響 `StrategyEvaluationService` 最後一步「進場價/平均價與現價 vs 均線」雙價格比較的通知結果，不得影響：官方收盤價擷取與 SQLite 寫入、MA5／MA20／MA60／MA120 計算與保存、鉅亨網交叉驗證、Excel 七欄均價前置輸出。兩個欄位同時無效時，`StrategyEvaluationService.Evaluate` 會分別產生 `AlertKind.EntryAveragePriceInvalid` 與 `AlertKind.CurrentPriceInvalid` 兩筆通知，兩個原因都必須讓使用者看見；`StrategyAlerts` 唯一鍵已加入 `AlertKind`，避免兩筆通知互相覆蓋。`DailyJobService.RunAsync` 會先以 `StockDailyPrice` 當日 DB 股票全集完成均線前置保存，再讀取 Excel 持股；客戶比對只讀取已保存的 `StockMovingAverage`。每日均價前置不呼叫歷史回補，DB 歷史不足時直接保存異常。`tests/YiHeLee.Tests/DailyJobServiceTests.cs` 以 Fake／Spy 建構完整依賴鏈驗證此行為並防止回歸。

## 逐檔歷史完整性驗證（2026-07-11 新增）

修正「市場整體交易日數已足夠、但個別股票仍不足」（例如上櫃市場整體已有 120 日，但 5351 鈺創、8299 群聯、3691 碩禾等個股只有 83／76／65 筆）導致只有 MA5、缺少 MA20／60／120 的問題：

- `StockIdentityResolutionService`（`YiHeLee.Application/Services/`）：Excel 讀出的原始股票代碼先經
  `StockCodeNormalizer` 正規化，再以官方股票主檔（`IMarketDataRepository.GetStockMarketTypesAsync`）
  驗證身分；查無資料且為 1～3 碼純數字時，嘗試補零為 4／5 碼後再查一次官方主檔，**只有補零後代碼確實
  存在於官方主檔才採用**，不盲目補零。回傳結果同時標示是否納入均線策略（`StockIdentityResolver` 依
  台股實際商品格式判斷，權證明確排除）。
- `MarketPriceService.BackfillHistoryAsync` 完成判斷改為「市場整體交易日數足夠」**且**「傳入的持股
  逐檔皆已達 `RequiredTradingDaysForMa120`」兩者皆成立才停止；回補本身仍是整批市場下載（效率不變），
  只有「是否已完成」的判斷改為逐檔確認。
- `IMarketDataRepository.GetDistinctTradeDateCountAsync(stockCode)`（既有方法，先前僅用於興櫃逐檔
  回補）擴大套用到上市／上櫃逐檔檢查，不需新增 Repository 方法。
- 回補批次若出現 `Failed` 狀態（官方來源逾時或格式異常），`DailyJobService` 會把該市場仍
  `InsufficientHistory` 的股票均線結果升級為 `CalculationStatus.BackfillFailed` 並附上原因，與「單純
  歷史資料不足（可能為近期上市）」明確區分，皆反映在 Excel 輸出「計算狀態」「缺少原因」欄位。

### 興櫃回補記憶化，避免結構性補不到的持股每次執行都重複查詢（2026-07-11 新增）

逐檔完成判斷要求興櫃持股也達到 `RequiredTradingDaysForMa120`，但興櫃很多持股是最近才開始交易，回補迴圈往回走到更早的日期時官方端點（`emerging/historical` 個股月份查詢）永遠查不到資料，逐檔完成條件因此永遠不會成立。實機資料庫比對發現：沒有記住「查過且確認沒有」的結果時，每一次執行（含當日重跑）都會對同一批股票、同一批過去日期重新發送請求，單次執行因此額外多花約 2 分鐘、逾 200 次請求，且天天重複發生。

修正：`MarketPriceService.BackfillHistoryAsync` 對興櫃持股逐日回補前，先呼叫 `IMarketDataRepository.GetConfirmedNoEmergingDataCodesAsync` 排除先前已確認查無資料的（股票代碼、交易日期）組合，只對真正還沒查過的組合發送請求；抓取後仍查無資料時呼叫 `RecordConfirmedNoEmergingDataAsync` 記錄，供未來執行略過（歷史資料一經確認即不會改變，新表 `EmergingHistoricalNoDataProbe`，見 `docs/03_資料庫結構.md`）。同時：三個市場當日皆因已足夠或已確認查無資料而略過抓取時，不再等待節流延遲。

實機驗證第一次／第二次執行發現，興櫃真實請求已降為 0 次，但上市／上櫃對「過去日期已確認休市」（例如國定假日）同樣沒有記住結果，每次執行仍會重新查詢（實測約 17 次）。對稱補上：`FetchAndSaveOneAsync` 於歷史回補時，先呼叫 `IMarketDataRepository.HasResolvedHolidayBatchAsync`（查詢既有 `OfficialPriceBatch.Status=Holiday`，不需新表），先前已確認休市的過去日期直接略過。詳見 `docs/05_異動紀錄.md`。

### 歷史回補耗盡記憶化，避免同一指定日期重跑反覆掃描（2026-07-11 新增）

逐檔完成判斷雖能避免市場整體 120/120 卻個股不足的誤判，但對近期掛牌、停牌或歷史來源確定查無資料的股票，整段回看範圍走完後仍可能永遠不足。過去同一策略日期重跑時，程式會重新從 `targetDate - 1` 往前掃一輪；即使每一天都因 DB 已有資料而沒有新增，畫面仍會持續顯示「已完成 yyyy-MM-dd 自動回補：新增 0、更新 0」，使用者看起來像同一天一直回補。

修正：`MarketPriceService.BackfillHistoryAsync` 於每輪仍先即時查詢 DB 逐檔筆數；若仍不足且先前已針對相同 `targetDate`、`RequiredTradingDaysForMa120`、`MaxBackfillLookbackCalendarDays` 與持股組合走完整個回看範圍，則直接回報「先前已走完整個回看範圍，本次略過重複回補」。首次走完整段且沒有 `Failed`／`NotPublished` 等暫時性官方狀態時，會寫入 `HistoricalBackfillExhaustionProbe`。若之後 DB 已因其他方式補足，因服務一律先重新計算逐檔筆數，仍會正常判定為足夠，不會只依記憶化紀錄跳過。

畫面文字也同步調整：歷史回補遇到某市場某日期 DB 已有正式收盤價時，單日摘要顯示「DB 已有資料，略過」，不再顯示成「新增 0 筆、更新 0 筆」造成誤解。

## 官方每日收盤價來源分層

- `ITwseMarketDataProvider`／`TwseMarketDataProvider`、`ITpexMarketDataProvider`／`TpexMarketDataProvider`：只呼叫官方端點、轉成 DTO、回報來源資料日期與解析錯誤，不含日期驗證或 SQL。
- `IMarketPriceService`／`MarketPriceService`：台北日期與來源日期驗證、休市／尚未公布判斷、冪等 Upsert、批次狀態記錄、歷史回補協調。
- `IMovingAverageService`／`MovingAverageService`：依有效交易日計算 MA5／MA20／MA60／MA120，資料不足時對應欄位為 `null`。
- `DailyMarketDataJob`／`HistoricalBackfillJob`：排程觸發用的薄封裝，只呼叫 `IMarketPriceService`，不含 HTTP、HTML／JSON 解析或 SQL。
- `IMarketDataRepository`／`SqliteMarketDataRepository`：`StockMaster`／`StockDailyPrice`／`StockMovingAverage`／`OfficialPriceBatch` 的參數化 SQL、交易與 Upsert。

## 重試層級

- 爬蟲短重試：網路、瀏覽器、DOM 暫時錯誤。
- Excel 短重試：RPC 呼叫被拒、Excel 忙碌、正在編輯儲存格。
- 排程長重試：網站尚未更新當日資料、短重試耗盡等可重試狀態。
- 設定無效、必要 Provider 不存在、Excel 唯讀／受保護檢視、輸出頁籤受保護等不可重試錯誤，需要人工修正。
- 非必要來源必須集中／店頭兩個市場都成功才會納入正式批次；其中一個市場失敗時整個非必要來源略過。
- 2026-07-11 更新：每日流程改為先處理官方收盤價，再以 DB 當日股票全集與 DB 既有歷史資料計算並保存 MA5／MA20／MA60／MA120；歷史不足直接保存 `InsufficientHistory` 並由 WinForms「均價資料異常」顯示，不在每日 Job 中自動回補。之後才擷取鉅亨網多頭／空頭排列清單作保存與交叉驗證，最後讀取 Excel 客戶持股並用已保存的 DB 均線結果做下游客戶比對。鉅亨網不再位於流程最前面，也不作正式均線來源。
- 2026-07-11 更新：TPEx 上櫃歷史回補改用新版 `afterTrading/otc` 端點與斜線日期格式，避免舊端點或壓縮日期格式靜默回傳最新交易日。興櫃歷史回補改用 TPEx 官方 `emerging/historical` 個股月份端點；因官方只提供「個股＋月份」查詢，程式只對 Excel 持股中已辨識為興櫃且 DB 缺資料的股票逐檔補。
- 2026-07-09 更新：鉅亨網多頭／空頭排列清單擷取改為 best-effort，不再是整批工作成功與否的門檻。原因是均線正式判斷已改依 TWSE／TPEx 官方資料計算，不再依賴鉅亨網數值；鉅亨網尚未更新或擷取失敗時，`DailyJobService` 僅記錄提醒訊息（`JobRunSummary.Message` 附加「提醒：…」字樣、`JobRunDetails` 仍完整記錄失敗細節），不影響已由官方價格算出的正式均線、策略判斷與 Excel 寫入。
- 官方每日收盤價（TWSE／TPEx）：來源資料日期不等於 `targetDate` 時記為 `NotPublished`（可重試，等同「網站尚未更新」）；TWSE 明確回報無資料時記為 `Holiday`。TPEx 端點對非交易日會靜默回傳最近一個交易日資料而非報錯，因此僅在同日 TWSE 已確認休市時，才會把 TPEx 的日期不符情形也記為 `Holiday`，否則一律視為 `NotPublished`，絕不接受用前一交易日資料頂替當日。
- 歷史回補以 DB 為準：若 `StockDailyPrice` 已保存指定市場、指定交易日的正式收盤價，即使沒有對應的歷史回補批次紀錄，也略過該日該市場的重複抓取。

## 歷史收盤價查詢畫面／使用者手動回補（2026-07-09 新增）

```text
TrayApplicationContext（選單「歷史收盤價」）
  -> HistoricalPriceForm（查詢畫面＋抓取進度面板）
     -> 查詢：IMarketDataRepository.QueryDailyPricesAsync（分頁＋SQL Window Function 計算 MA5/20/60/120）
     -> 立即回補：IStockHistoryImportService.CreateJobAsync（快速建立 StockPriceImportJob／Task，皆為 Queued）
        -> IStockHistoryImportService.RunJobAsync（背景執行，不阻塞 UI 執行緒）
           -> Parallel.ForEachAsync（MaxConcurrency 上限，預設4，不超過8）
              -> IMarketPriceService.FetchAndSaveSingleAsync（單一市場＋單一交易日，重用既有驗證與冪等 Upsert）
              -> 暫時性錯誤指數退避重試（2/4/8秒，經 TransientFailureClassifier 區分是否可重試）
           -> IStockPriceImportRepository（每個工作完成後即時彙總進度，供輪詢與重新整理回復）
     -> 取消抓取：CancellationTokenSource.Cancel -> IStockPriceImportRepository.CancelJobAsync
     -> 與鉅亨網交叉驗證：CrawlerRegistry（既有鉅亨爬蟲）-> IStockPriceValidationService.ValidateAsync
        -> IStockPriceValidationRepository（CnyesCrossValidation，僅供驗證追查，不覆蓋官方資料）
```

- 工作單位固定為「市場＋交易日期」，例如回補 5 個有效交易日、市場範圍全部時，約產生 10 個工作，而非每檔股票各發送一次請求。
- 候選交易日由觸發當下台北日期往前一日開始，排除週六、週日；國定假日等無法事先得知的休市日仍會實際送出請求，由既有官方來源誠實回報並記為 `Holiday`，不計入有效交易日、也不算失敗。
- 進度（`StockPriceImportJob`／`StockPriceImportTask`）皆彙總自資料庫明細列，不依賴記憶體計數器，重新整理畫面或重啟程式後仍可回復目前進度。
- SQLite 並行安全：沿用既有 `SqliteMarketDataRepository` 慣例（每次操作各自開啟連線＋`busy_timeout=10000`＋單一交易），並行只發生在 HTTP 抓取階段，資料庫寫入仍是「每個市場＋交易日完整下載、解析、驗證後才進入單一交易 Upsert」，不會有多個 Task 共用同一個 `DbConnection`。

## 權限

預設與使用者同權限執行，以連接一般權限的 Excel。可從系統匣選單要求 UAC 提升，但提升後 Excel 也必須以相同權限啟動。
