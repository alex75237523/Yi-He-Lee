# 異動紀錄

## 2026-07-13（修正：盤中完成後顯示每日結果，並顯示執行進度）

- 修正 `TrayApplicationContext.OnIntradayRunCompleted`：盤中判斷成功或部分成功時，結果頁資料一律由 `summary.Alerts` 完整清單轉成既有「每日結果」顯示資料，不再用 `NewNotificationCount > 0` 作為是否切到結果頁的唯一條件。
- 手動「立即執行盤中判斷」即使 `NewNotificationCount=0`，只要本次完成成功／部分成功，也會啟動原本「每日結果」顯示流程；因此「目前成立 9 筆、新通知 0 筆」仍會在結果頁看到 9 筆成立清單。

- 盤中執行中新增狀態文字與進度條：開始執行、讀取設定、解析上一交易日均價基準、讀取已保存均價、讀取 Excel 客戶持股與最新現價、正規化股票代碼、比對均價條件、更新通知去重狀態、完成／失敗皆會透過既有 `IUserInteraction.ShowStatus` 更新「操作」頁進度條。
- `IntradayRunSummary` 新增 `IsManualRun`，供 UI 區分手動執行與自動 Tick 的顯示策略；不影響資料庫 Schema、策略判斷公式、收盤更新流程或 Excel「每日五日均價策略」七欄輸出。
- 測試新增／更新：`IntradayMonitoringServiceTests` 驗證盤中執行會回報狀態文字與進度百分比；`TrayApplicationContextIntradayDisplayTests` 驗證手動盤中 `NewNotificationCount=0` 仍顯示每日結果，且 `ToJobRunSummary` 使用完整 `summary.Alerts` 而不是只顯示 `NewlyTriggeredAlerts`。

## 2026-07-13（盤中基準準備完成後同次判斷與後續快速路徑）

- `IntradayMonitoringService` 在上一交易日基準缺漏時不再直接結束；改為呼叫 `BaselinePreparationService.EnsureBaselineDataAsync` 補齊缺漏交易日官方收盤價、重算並保存 MA5／MA20／MA60／MA120，接著重新呼叫 `TradingDateResolver` 驗證基準。驗證成功後，同一次呼叫內繼續 `ReadHoldingsAsync -> Evaluate -> IntradayEvaluationRun`，不要求使用者回補後再按第二次。

- 測試新增／更新：`IntradayMonitoringServiceTests` 覆蓋缺基準同次判斷、Ready 快速路徑、準備失敗不讀 Excel、準備中略過；`IntradayMonitoringSqliteBaselineTests` 使用實際 SQLite 驗證第一次空庫「資料缺漏 → 回補 → 均價完成 → 讀 Excel → 判斷」，以及第二次與模擬重啟後「沿用已完成基準 → 讀 Excel 最新價格 → 判斷，不再回補」。`dotnet test` 通過 324／324。

## 2026-07-13（fix：均價高於任一價格即觸發通知，正式更正比較方向）

分支：`fix/reverse-dual-price-moving-average-condition`。

**本次是正式的比較方向更正，不是單純調整程式寫法。** 2026-07-12 曾記載並實作「進場價/平均價與現價同時大於或等於均價」才觸發；此規則已由本次更正取代。現行規則如下：

```text
舊條件（2026-07-12 歷史規則，已作廢）：
    進場價/平均價 >= 均價
    且
    現價 >= 均價

新條件（2026-07-13 現行正式規則）：
    均價 >= 進場價/平均價
    或
    均價 >= 現價
```

- MA5 成立 = MA5 >= 進場價/平均價 **或** MA5 >= 現價。
- MA20 成立 = MA20 >= 進場價/平均價 **或** MA20 >= 現價。
- MA120 成立 = MA120 >= 進場價/平均價 **或** MA120 >= 現價。
- MA5／MA20／MA120 之間仍為 OR，任一均價成立就通知。
- MA60 繼續保存及顯示，不參與通知。
- 「進場價/平均價」與「現價」仍為兩個獨立欄位；任一欄位無效時不得以另一欄、官方收盤價、昨日資料或預設值代替，原有「進場價／平均價異常」與「現價異常」通知機制保留。

**程式修改**：

- `StrategyEvaluationService.IsDualPriceMatch` 改為 `ma >= entryAveragePrice || ma >= currentPrice`；`Evaluate` 與 `EvaluateAll` 繼續共用同一核心方法，避免兩處判斷不一致。
- `StrategyAlert.TriggerDescription` 與 `HoldingStrategyResult.TriggerDescription` 改為「均價已大於或等於進場價/平均價或現價其中一項」；未觸發訊息改為「MA5／MA20／MA120 尚未有任一均價大於或等於進場價/平均價或現價」。
- `MainForm.Results` 摘要改為「符合均價高於任一價格條件」；判斷明細改為均價在比較左側，例如「MA20：均價 480 >= 進場價/平均價 470；均價 480 < 現價 500；至少一項條件成立」。
- `database/schema.sql`、`Entities.cs` 與測試註解同步調整語意；資料庫 Schema 無需變更。

**測試修改**：

## 2026-07-12（歷史：進場價與現價雙重均價條件，已由 2026-07-13 正式更正取代）

分支：`fix/dual-price-moving-average-condition`。

> 注意：本章保留歷史異動脈絡，其中「進場價/平均價與現價同時大於或等於均價」及相關畫面文字已不是現行規則。現行規則請以 2026-07-13 條目為準：`均價 >= 進場價/平均價` 或 `均價 >= 現價` 其中一項成立即通知。

**問題根因**：2026-07-11 曾記載「原『進場價／平均價』為口誤，正確欄位為『現價』」，並將客戶比對的比較基準由雙欄位改為單一「現價」欄位。此更正本身是錯誤的——實際客戶工作簿（`進場日／代號／股名／進場價/平均價／張數／現價`）確實同時存在「進場價/平均價」與「現價」兩個獨立欄位，並非同一欄位誤植。使用者正式提出需求變更，要求恢復並強化雙價格判斷：每一條均價都必須「進場價/平均價」與「現價」同時大於或等於該均價才算成立，禁止只取其中一個判斷、禁止混用、禁止以其他資料（官方收盤價、昨日現價、對方欄位、0、任意預設值）代替任一價格。

**新舊邏輯對照**：

```text
2026-07-11 舊版（本次取代）：
    MA5／MA20／MA120 成立 = MA <= 現價（單一欄位）

本次新版（2026-07-12 恢復並強化）：
    MA5成立  = 進場價/平均價 >= MA5  且 現價 >= MA5
    MA20成立 = 進場價/平均價 >= MA20 且 現價 >= MA20
    MA120成立 = 進場價/平均價 >= MA120 且 現價 >= MA120
    整體條件成立 = MA5成立 或 MA20成立 或 MA120成立
    MA60 僅保存與顯示，不參與觸發。
```

### 最高優先規則（本次強化並補齊自動測試）

Excel「現價」欄位串接外部 DDE（看盤軟體），可能出現 `#N/A`、`#NAME?`、`#GETTING_DATA`、`#CONNECT!`、`#BLOCKED!`、空白、0、負數、無法解析文字，或看盤軟體未開啟／DDE 尚未連線等情形。這些情形**只能**造成「現價狀態：無效」「最終策略比較：無法判斷」，絕對不可以：提前 `return`、跳過該股票官方收盤價處理、跳過歷史回補、不計算均線、把已算出的均線改成 `null`、把整批 Job 標記失敗、阻擋其他客戶或其他股票、用收盤價或昨日價格冒充現價、用其他數字代替 DDE 現價。

審閱 `DailyJobService.RunAsync` 既有流程後確認：官方收盤價擷取（`DailyMarketDataJob`）、歷史回補（`HistoricalBackfillJob`）、均線計算（`MovingAverageService`）、鉅亨網交叉驗證皆在讀取 Excel 持股（含 DDE 現價）之後，且皆不依賴 `CustomerHolding.CurrentPrice`；`StrategyEvaluationService` 對 DDE 無效持股也已個別產生 `AlertKind.CurrentPriceInvalid` 通知並照常帶入已算出的均線欄位，不會讓官方流程中斷。本次新增 `tests/YiHeLee.Tests/DailyJobServiceTests.cs`（以 Fake／Spy 建構完整 `DailyJobService` 依賴鏈，不需真實 Excel／SQLite／HTTP）以自動測試明確驗證此既有正確行為並防止未來回歸，涵蓋：`#N/A`／`#NAME?`／空白／0／負數／無法解析文字六種 DDE 無效情況官方價格與均線仍正常完成；同一批持股中單一股票 DDE 錯誤不影響其他股票；DDE 無效但均線資料完整（120 日）時全部均線正常顯示且不得標記「均線計算失敗」；讀取持股→回補→均線計算→鉅亨驗證→Excel 寫入的執行順序驗證；Excel 完整結果列數等於有效持股數而非 alerts 數量。

分支：`fix/per-stock-moving-average-history`。修正「每日五日均價策略」中大量股票只有 MA5、缺少 MA20／MA60／MA120，以及非股票資料被誤判的問題。

### 根因

1. **歷史回補以市場整體交易日數判斷是否足夠**：`MarketPriceService.BackfillHistoryAsync` 原本只檢查
   `GetDistinctTradeDateCountAsync(marketType)`（該市場「任一股票」湊出的相異交易日數），只要市場整體達
   120 日就停止回補；但個別股票（例如上櫃 5351 鈺創、8299 群聯、3691 碩禾）可能因掛牌較晚、曾停牌或
   回補當時尚無資料，實際筆數遠低於 120。市場整體足夠不代表每一檔股票都足夠。
2. **股票代碼正規表示式過寬**：`^[0-9A-Z]{4,10}$` 會把 Excel 中的權益數、保證金等 7～8 位數金額
   （例如 `10037677`）誤判為股票代碼。
3. **ETF 前導零遺失時無回復機制**：Excel 儲存格若被當成數字讀取，`0050` 會讀成 `50`；原程式沒有任何
   補零機制，也沒有官方主檔比對，這類持股會直接判斷失敗或被過寬正規表示式誤收。
4. **Excel 表格邊界辨識只認「現價」**：無法識別「現貨現價」等同義表頭；且只有遇到「新的持股表頭」或
   「出場價表頭」才會結束區塊，遇到「日期／權益數／保證金／小計」等其他彙總表格時不會停止，會把該表格
   內容誤判為持股列延伸掃描。

### 修正方式

- **`MarketPriceService.BackfillHistoryAsync`**：新增可選參數 `listedStockCodes`／`otcStockCodes`
  （既有 `emergingStockCodes` 沿用），完成判斷改為「市場整體交易日數足夠」**且**「傳入的持股逐檔皆已
  達 `RequiredTradingDaysForMa120`」兩者皆成立才停止回補；任一持股逐檔仍不足，即使市場整體已足夠也會
  繼續回補該市場（回補仍是整批市場下載，效率不變，只有「完成判斷」改為逐檔）。未傳入代碼時維持既有
  「僅檢查市場整體」行為，向下相容。`IMarketPriceService.BackfillHistoryAsync`、`HistoricalBackfillJob.RunAsync`
  同步新增對應參數並轉發。
