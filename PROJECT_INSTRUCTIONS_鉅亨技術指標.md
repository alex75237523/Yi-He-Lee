# 專案固定指令：鉅亨網台股技術指標每日擷取

本文件為此專案所有 AI、Codex、Claude Code 與人工開發的強制規範。開始任何修改前，必須先閱讀目前程式碼、資料庫結構、既有排程與文件，再依本文件執行；不得未確認現況就直接重寫或更換技術棧。

> **2026-07-09 更新（正式均價來源改為官方資料）：** 以下第一～八節（原始鉅亨網擷取規範）仍為歷史需求並持續適用於「多頭／空頭排列清單擷取」；但均線策略正式判斷與 Excel 輸出的 MA5／MA20／MA60／MA120 數值，**改由臺灣證券交易所（TWSE）與證券櫃檯買賣中心（TPEx）官方每日收盤價，由本系統依有效交易日自行計算**，鉅亨網兩來源不再作為正式均價來源，僅保留作多頭／空頭排列完整清單保存與交叉驗證。詳見本文件最末新增之「九、官方每日收盤價與均線（TWSE／TPEx）」一節。

## 一、每日資料擷取需求

### 1. 固定執行時間

- 時區一律使用 `Asia/Taipei`。
- 每日台北時間 `13:35`，台股收盤後執行。
- 排程時間、查詢日期、資料日期與紀錄時間不得使用伺服器未轉換的 UTC 日期。
- 必須由程式明確取得台北當地日期，例如 `targetDate`，不可硬編日期。

### 2. 固定資料來源

每日必須抓取以下兩個網址的完整 List 清單：

1. `https://www.cnyes.com/twstock/a_technical4.aspx`
   - 資料類型：股價多頭排列。
2. `https://www.cnyes.com/twstock/a_technical5.aspx`
   - 資料類型：股價空頭排列。

兩個頁面都必須抓取：

- 集中市場。
- 店頭市場。

不得只抓預設市場、畫面第一批資料、前 50 筆或部分清單。必須確認該日期、該市場的表格所有資料列都已完成擷取。

### 3. 當日日期強制規則

- 「設定查詢」日期一定要使用排程執行當日的台北日期。
- 頁面顯示的資料日期必須與 `targetDate` 完全一致。
- 請求日期、頁面日期、資料寫入的交易日期必須一致。
- 絕對禁止因網站尚未更新、休市、例假日或錯誤而自動改抓前一交易日。
- 絕對禁止把前一天資料寫成今天。
- 在頁面日期尚未等於當日以前，不得寫入正式資料表。
- 若 13:35 時網站仍顯示前一交易日，必須記錄「網站尚未更新當日資料」並依既有排程重試機制重試。
- 若為休市日或當日確實沒有資料，應記錄明確狀態，不得用舊資料補齊。

### 4. 擷取欄位

每一筆至少保存以下資料：

- 查詢／交易日期。
- 資料類型：`BullishAlignment`（多頭排列）或 `BearishAlignment`（空頭排列）。
- 市場別：集中市場或店頭市場。
- 股票代碼。
- 股票名稱。
- 收盤價。
- 5 日均價。
- 20 日均價。
- 60 日均價。
- 120 日均價。
- 來源網址。
- 擷取開始時間。
- 擷取完成時間。
- 建立時間與最後更新時間。

數值欄位應使用適合的 Decimal 型別，不得以顯示文字永久保存為唯一資料來源。

### 5. 完整性與日期驗證

寫入前必須完成以下檢查：

- HTTP 回應成功。
- 頁面不是錯誤頁、驗證頁或不完整 HTML。
- 頁面資料類型與來源網址相符。
- 市場別與本次查詢條件相符。
- 頁面顯示日期等於 `targetDate`。
- 表頭結構存在，欄位可正確對應。
- 每筆股票代碼、名稱與必要數值格式有效。
- 完整讀取所有資料列後，才可進入資料庫交易。
- 空清單必須能區分「合法零筆」與「解析失敗／網頁尚未更新」，不可直接視為成功。

任何一項驗證失敗，都不得把該批次標記為成功。

## 二、資料庫與重跑規則

### 1. 關聯式與去重設計

- 資料庫必須採關聯式設計，避免重複保存股票基本資料。
- 股票基本資料應以股票代碼或既有股票主檔關聯，不得因每日擷取重複新增相同股票名稱。
- 建議資料唯一鍵至少包含：
  - `TradeDate`
  - `IndicatorType`
  - `MarketType`
  - `StockCode`
- 排程必須具備冪等性，重跑同一天、同資料類型、同市場時不得產生重複資料。
- 寫入採用安全的 Upsert、先比對後更新，或符合現有架構的等效做法。
- 同一批次應使用資料庫交易，避免只寫入部分市場或部分資料列。

### 2. 資料表與欄位

- 新增的每個資料表與欄位都必須有繁體中文註解。
- 不得為方便而將整份頁面資料塞入單一 JSON 欄位取代正常關聯式設計。
- 大量資料處理可使用暫存表、中繼表或快取表，但必須有清除與失敗復原機制。
- SQL 必須參數化，不得拼接外部網頁內容或使用者輸入。
- `ALTER`、`UPDATE`、`DELETE` 必須說明條件與影響範圍。

## 三、程式架構規範

### 1. 分層責任

依目前專案技術棧與既有架構實作，不得為此功能任意重寫整個專案。C# 專案須遵守 MVC／三層架構精神：

- 排程層：只負責在 13:35 觸發工作，不放網頁解析與資料庫商業邏輯。
- Service／Business Layer：負責日期規則、流程控制、重試判斷、批次驗證、交易一致性與 Upsert 規則。
- Crawler／Provider／Adapter：只負責請求指定來源、切換日期與市場條件、取得原始內容。
- Parser：只負責將頁面表格轉成結構化資料，並回報解析錯誤。
- Repository／DAO：只負責 CRUD、Query、參數化 SQL 與資料轉換。
- Controller：若有管理介面，只處理 HTTP request、Claims、呼叫 Service 與回傳 View／JSON。
- ViewModel／ApiModel／Entity 不得混用職責。
- 前端只負責查詢畫面、狀態與操作回饋，不得成為唯一的日期或資料驗證來源。
- 後端 Service 必須重新驗證所有重要規則。

### 2. 共通程式規則

- 使用物件導向、低耦合、高內聚。
- 可重複使用規則放在 Service 或 Helper，不得複製到多個排程、Controller、View 或 JavaScript。
- 使用既有 HttpClient、排程、Logging、DI 與設定機制；沒有充分理由不得另外導入重型框架。
- 網址、逾時、重試條件與排程時間應設定化，但上述兩個固定來源與「當日日期」規則不得被任意關閉。
- HTTP 呼叫必須設定合理 Timeout、CancellationToken、錯誤處理與識別用 User-Agent。
- 不得繞過網站安全機制、驗證或存取限制。
- 所有重要程式碼必須有繁體中文註解。

## 五、測試與驗收

至少涵蓋以下測試：

- 多頭排列頁面解析。
- 空頭排列頁面解析。
- 集中市場與店頭市場解析。
- 當日日期正確時可寫入。
- 頁面仍為昨日日期時拒絕寫入。
- 休市或合法零筆資料的判斷。
- HTML 欄位順序改變或缺欄時能失敗並留下紀錄。

- 某一市場失敗時整批交易與狀態正確。
- 時區跨日、UTC 與台北日期不同時仍使用台北日期。

修改後應執行現有 Solution 的 Build 與相關測試。無法執行時，必須說明原因並提供人工驗證步驟。

## 九、官方每日收盤價與均線（TWSE／TPEx）（2026-07-09 新增）

### 1. 資料來源分工

- **鉅亨網**（`a_technical4.aspx` 多頭排列、`a_technical5.aspx` 空頭排列）：僅作多頭／空頭排列完整清單（集中＋店頭）保存與交叉驗證，**不再是正式均價來源**，不可刪除或停用。
- **TWSE**（臺灣證券交易所）：上市股票官方每日收盤價。實測採用 `https://www.twse.com.tw/exchangeReport/MI_INDEX?response=json&date={yyyyMMdd}&type=ALLBUT0999`（可指定西元日期，回應含明確 `date` 欄位供驗證；非交易日回應 `stat` 明確為「查無資料」，無 `date` 欄位）。`openapi.twse.com.tw` 之 `STOCK_DAY_ALL` 雖有逐筆 `Date` 欄位可驗證，但不接受日期參數、僅回傳最新一筆，故改用／輔以上述可指定日期端點供每日驗證與歷史回補共用。
- **TPEx**（證券櫃檯買賣中心）：上櫃股票官方每日收盤價。實測採用 `https://www.tpex.org.tw/web/stock/aftertrading/daily_close_quotes/stk_quote_result.php?l=zh-tw&d={民國年/月/日}&s=0,asc,0`。**重要實測發現**：此端點對非交易日或未來日期查詢，會靜默改回傳「最近一個有交易資料的日期」，HTTP 200、有資料列，但回應中的 `date` 與表格內 `date` 皆為該較早日期，**不會**回報錯誤或明確休市訊息。因此程式一律比對回應 `date` 是否等於 `targetDate`，不符時視為 `NotPublished`（除非同日 TWSE 已明確確認休市，才記為 `Holiday`），絕不可只憑 HTTP 200 或有資料列視為當日成功。`openapi.twex...` 的 `tpex_mainboard_daily_close_quotes` 同樣僅回傳最新一筆、不支援日期參數，故改用上述可指定日期端點。
- **系統**：MA5／MA20／MA60／MA120 一律由本系統依 TWSE／TPEx 官方收盤價、依「有效交易日」（非日曆日）自行計算，中間過程不四捨五入，資料不足時對應欄位為 `null`。

### 2. 分層與資料表

詳見 `docs/02_架構與資料流程.md`「官方每日收盤價來源分層」與 `docs/03_資料庫結構.md`「官方每日收盤價與均線」：`ITwseMarketDataProvider`／`ITpexMarketDataProvider`、`IMarketPriceService`、`IMovingAverageService`、`DailyMarketDataJob`／`HistoricalBackfillJob`、`IMarketDataRepository`（`StockMaster`／`StockDailyPrice`／`StockMovingAverage`／`OfficialPriceBatch`）。

### 3. 日期與休市規則

與第一節鉅亨網規則相同精神：請求日期、來源回報日期、資料庫 `TradeDate`、`targetDate` 四者必須一致才可寫入正式資料；休市或尚未公布時記錄明確狀態並依設定化重試機制重試，絕不回抓前一交易日或用前一交易日資料補齊。歷史回補（`HistoricalBackfillJob`）與每日正式排程（`DailyMarketDataJob`）分開執行、分開記錄批次狀態；歷史回補只補建 `targetDate` 以前的資料，一律保存原始 `TradeDate`。

### 4. 鉅亨網與官方流程解耦（2026-07-09 更新）

均線正式判斷已不再依賴鉅亨網數值，因此鉅亨網多頭／空頭排列清單擷取（第一～八節）改為 **best-effort**：尚未更新當日資料或擷取失敗時，`DailyJobService` 只記錄提醒訊息並繼續執行官方價格擷取、均線計算、策略判斷與 Excel 寫入，不再讓鉅亨網單一來源阻擋整批工作。`JobRunDetails` 仍完整記錄鉅亨網每次失敗細節供人工追查；`OfficialPriceBatch` 則仍依第九節規則嚴格驗證來源日期，任一官方來源尚未公布或失敗時，整批工作依然視為可重試失敗，不得寫入策略結果。
